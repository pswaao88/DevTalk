# DESIGN-003-Resolved기준정의

## 목적
DevTalk에서 Resolved가 가지는 의미를 명확히 정의하고,  
자동화가 아닌 **사람의 판단**을 기준으로 문제 해결 종료를 선언하는 기준을 정리한다.

Resolved는 문제를 해결했는가가 아니라,  
**이 사고 흐름에 대한 판단을 여기서 종료하겠다**라는걸 의미한다.

## 내용

### Resolved의 의미
- Resolved는 문제 해결에 대한 **사람의 판단 종료 선언**을 의미한다.
- Resolved는 자동 분석이나 AI 판단에 의해 결정되지 않는다.
- Resolved는 문제의 성공·실패 여부를 판정하는 상태가 아니다.

### Session 상태 정의
Session은 다음 두 가지 상태 중 하나를 가진다.
- **ACTIVE**: 사고 흐름이 진행 중인 상태
- **RESOLVED**: 현재 시점에서 판단을 종료한 상태

Session 상태는 현재의 판단 상태를 나타내며,  
과거의 판단 이력을 의미하지 않는다.

### Resolved 적용 방식
- 사용자는 특정 Session에 대해 직접 Resolved를 확정한다.
- Resolved 확정 시 변경되는 것은 Session의 상태뿐이다.
- Resolved 확정과 동시에 새로운 Message를 자동으로 생성하지 않는다.

### Resolved 토글
- Session은 `ACTIVE ↔ RESOLVED` 상태 전환이 가능하다.
- 사용자는 Resolved 이후에도 다시 Active(Unresolved) 상태로 되돌릴 수 있다.
- 상태 전환은 여러 번 발생할 수 있으며, 이는 자연스러운 개발 흐름으로 간주한다.

### Resolved 이후 채팅
- Session이 RESOLVED 상태이더라도 메시지 추가는 가능하다.
- Resolved는 채팅을 제한하는 잠금 상태가 아니다.
- Resolved 이후에 이어진 대화 역시 동일한 사고 흐름의 연장으로 기록된다.

### Resolved와 로그의 관계
- Resolved / Unresolved 상태 전환은 중요한 판단 이벤트이다.
- 모든 상태 전환은 세션의 로그 타임라인에 기록되어야 한다.
- 상태 변경은 SYSTEM 이벤트 메시지로 기록될 수 있다.
- Resolved는 기록을 종료하는 신호이지, 기록의 완성을 의미하지 않는다.

## 결과
위 기준을 통해 다음을 보장한다.

- 문제 해결 여부에 대한 최종 책임이 사람에게 있다.
- 판단이 바뀌는 과정 자체가 기록 자산으로 남는다.
- 자동 판단으로 인한 기록 왜곡을 방지한다.
- 해결되지 않았거나 재발한 문제도 자연스럽게 기록할 수 있다.
