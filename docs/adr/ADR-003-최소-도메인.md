# ADR-003: 최소 도메인 구성(Session / Message)

## 상태
Accepted

## 배경
초기부터 많은 도메인 엔티티를 도입하면,
구현 부담이 커지고 실제 사용까지 시간이 지연된다.

2회차의 목표는 구조 완성이 아니라,
"실제로 채팅이 되고 기록이 남는 상태"를 만드는 것이다.

## 결정
2회차에서는 다음 두 개의 핵심 도메인만 사용한다.

- Session: 문제 단위
- Message: 대화 및 기록 단위

Attempt, Resolution 등의 세분화된 도메인은 도입하지 않는다.

## 이유
- Message 하나로 질문, 응답, 실패 로그를 모두 표현할 수 있다.
- 구조가 단순해 구현 속도가 빠르다.
- 이후 필요해질 경우 도메인을 분리해도 확장이 가능하다.

## 결과
- Message에 role(USER/ASSISTANT/SYSTEM)과 status(SUCCESS/FAILURE)를 포함한다.
- 실패도 Message로 기록한다.
