# ADR-014: 계획변경 

## 상태
Accepted

## 결정 시점
2026-01-13

## 배경
초기 WarruruLab / DevTalk 프로젝트는  
DevTalk → ProblemWarruru → MCP → 고도화까지  
약 8주에 걸쳐 단계적으로 설계·구현·정리하는 계획을 전제로 수립되었다.

그러나 세미나 일정이 확정되면서,
- 2월 6일까지 모든 기능을 완성·배포·안정화해야 하고
- 비전공자 대상 전체 세미나에서 실제 동작하는 결과물을 설명해야 하는 상황이 되었다.

기존 8주 계획을 유지할 경우,
- 중간 단계의 설계만 존재하고
- 끝까지 이어지는 흐름을 완성하지 못한 상태로 발표하게 될 위험이 컸다.

이에 따라 “완성 가능한 범위”를 재정의하고,
계획 자체를 축소·압축하는 결정이 필요해졌다.

## 결정
기존 8주 개발 계획을 유지하지 않고,  
**2월 6일까지 완료를 목표로 하는 4주 압축 개발 플랜으로 전환한다.**

이 4주 플랜은 다음을 목표로 한다.

- DevTalk과 ProblemWarruru를 하나의 흐름으로 완성
- 배포된 환경에서 실제 시연 가능
- 성공·실패·판단이 모두 로그로 남는 구조 증명
- 비전공자에게 프로젝트 철학과 구조를 설명 가능

기간 내 완성을 방해하는 요소들은  
의도적으로 2/6 이후 고도화 항목으로 이월한다.

## 이유
- 일정 내 “끝까지 동작하는 제품”을 만드는 것이 최우선 과제가 되었기 때문이다.
- 완성되지 않은 고도화보다, 명확히 완주한 흐름이 발표·포트폴리오 측면에서 더 가치 있다고 판단했다.
- DevTalk의 핵심 철학(판단과 실패의 기록)은 DB·MCP 고도화 없이도 충분히 증명 가능하다.
- 압축된 일정에서는 범위 통제가 곧 품질 관리라고 판단했다.

의도적으로 포기하거나 미룬 항목은 다음과 같다.

- 정식 DB 스키마 고도화 및 마이그레이션 전략
- MCP 전면 적용 및 자동화 흐름
- 로그 품질 자동 평가/추천 로직
- UX 고급 기능(검색, 필터, 버전 관리 등)

## 결과
- 개발 로드맵이 “단계적 확장”에서 “완주 중심”으로 전환되었다.
- 4주 동안의 작업은 모두 배포·시연 가능 상태를 기준으로 진행된다.
- DevTalk → ProblemWarruru → 블로그 초안 생성까지의 최소 흐름이 완성된다.
- 이후 고도화 항목(DB, MCP, UX, 운영)은 2/6 이후 별도 계획으로 진행한다.
- 계획 변경 자체가 하나의 설계 판단 기록으로 남아, 프로젝트 문서와 발표 자료의 근거가 된다.

이 결정은 이후 일정 관리, 기능 범위 설정, 발표 구성에 직접적인 영향을 준다.
